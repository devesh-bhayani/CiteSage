# Pre-tuning baseline (archived)

These three files were generated by the **first** run of the Phase 3
evaluation harness on Ollama (qwen3:8b generator + qwen3-small grader),
before the Phase 3 tuning fixes landed in commit `a401efb`. They are
preserved for historical comparison only — **do not** treat them as the
current baseline.

| File                    | Origin                                         |
| ----------------------- | ---------------------------------------------- |
| `baseline_scores.json`  | aggregate metrics from the Ollama run          |
| `baseline_errors.jsonl` | per-query failure log from the Ollama run      |
| `eval_errors.jsonl`     | a separate (earlier) error log of partial runs |

## Why they're misleading now

All four targets failed:
- accuracy 66.9 % vs 85 %
- citation precision 23.6 % vs 90 %
- decline recall 50 % vs 85 %
- p95 latency 205.9 s vs 5 s

The bulk of those misses were code/measurement bugs that have since
been fixed:

1. The grade-relevance JSON parser silently kept *all* chunks on parse
   failure, so the grader was a no-op on the local model. Fixed in
   [src/citesage/graph/nodes.py](../../../src/citesage/graph/nodes.py).
2. Citation precision was measured as a strict-subset boolean per
   query (a query with 9 correct + 1 wrong cite scored 0). The
   harness now uses the spec-defined mean per-query
   `|cited ∩ expected| / |cited|`. Fixed in
   [src/citesage/evaluation/run_eval.py](../../../src/citesage/evaluation/run_eval.py).
3. Routing thresholds were too permissive (`confidence_threshold`
   0.7, `decline_threshold` -5.0). Tightened to 0.8 and -3.0.
4. Three queries failed entirely with Ollama connection errors
   because the client had no timeout. Fixed in
   [src/citesage/utils/llm_factory.py](../../../src/citesage/utils/llm_factory.py).
